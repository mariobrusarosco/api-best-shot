# Almanac Database Schema

## Status And Authority

This document contains only accepted database schema for the new Almanac application. Broad ideas
remain in `docs/backend-contract-notes.md`; open and proposed decisions remain in slice documents.

Drizzle schema declarations and generated migrations are the executable implementation of this
document. A mismatch between them is an error to resolve before continuing.

## PostgreSQL Boundary

All Almanac-owned tables use the `almanac` PostgreSQL schema.

## Explicit Non-Persistence Decision

The Almanac Contents section metadata is backend configuration owned by the Contents service. It
does not have a PostgreSQL table, Drizzle schema declaration, repository, or migration. The service
uses the Editions service's public count behavior only to derive the Teams index page number.

## `almanac.world_cup_editions`

Owner: Editions domain.

The host flag field is implemented by migration `0004_almanac_edition_host_flag_asset_key`.

| Column | Type | Null | Rules |
| --- | --- | --- | --- |
| `id` | `uuid` | No | Primary key; generated by PostgreSQL |
| `source_key` | `text` | No | Unique stable source identity |
| `year` | `smallint` | No | Unique; between 1930 and 2100 |
| `name` | `text` | No | Edition display name |
| `host_display_name` | `text` | No | Current accepted host display value |
| `host_flag_asset_key` | `text` | Yes | Provider-neutral host-country flag key used by the Editions index |
| `logo_asset_key` | `text` | Yes | Provider-neutral public asset key |
| `created_at` | `timestamptz` | No | Defaults to `now()` |
| `updated_at` | `timestamptz` | No | Defaults to `now()` |

The Editions index orders rows by `year DESC`.

The API derives these values instead of storing them:

```text
path         = /editions/{year}
pageNumber   = first edition page + ordered array index
host.flagUrl = ASSET_BASE_URL + host_flag_asset_key
```

`logo_asset_key` remains edition-logo data for other Almanac contexts. The Editions index does not
return it because that screen displays the host-country flag.

## `almanac.national_teams`

Owner: Teams domain.

Implemented by migration `0003_almanac_national_teams`.

This table represents reusable national-team identity, such as Brazil or Argentina. It does not
represent an edition-specific player roster.

| Column | Type | Null | Rules |
| --- | --- | --- | --- |
| `id` | `uuid` | No | Primary key; generated by PostgreSQL |
| `code` | `varchar(3)` | No | Unique national-team code, such as `BRA` |
| `display_name` | `text` | No | Public team name, such as `Brazil` |
| `flag_asset_key` | `text` | Yes | Provider-neutral public flag key |
| `created_at` | `timestamptz` | No | Defaults to `now()` |
| `updated_at` | `timestamptz` | No | Defaults to `now()` |

The Teams index orders rows by `display_name ASC`.

The API derives these values instead of storing them:

```text
path       = /teams/{lowercase code}
pageNumber = first team page + ordered array index
flagUrl    = ASSET_BASE_URL + flag_asset_key
```

## Current Relationships

No accepted relationship exists yet between `national_teams` and `world_cup_editions`.

Edition participation, tournament-specific squads, players, and appearance aggregation require a
future accepted slice and are not represented in the current schema.
